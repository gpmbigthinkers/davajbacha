# AGENTS.md — davajbacha

Operating rules and design laws for AI agents and human contributors working in this repository.

---

## 1. Security — Non-negotiable

### 1.1 Never publish secrets to Git

- **NEVER** commit, push, paste, diff, or otherwise expose the contents of `.env` (or any `*.env`, `.env.local`, `.env.*.local`).
- **NEVER** commit a real `DATABASE_URL`, connection string, API key, auth secret, admin password, or any credential — in any file, in any commit, in any branch, in any PR description, screenshot, log, or test fixture.
- The only env file allowed in the repo is `.env.example`, and it must contain **placeholders only** (e.g. `postgresql://user:password@host:5432/database`, `change-me-in-production-32-char-secret`). Never put real values there.
- If you accidentally see a real secret in a file, **stop and warn the user** before doing anything else. Do not echo it back, do not write it into logs, do not include it in a commit.

### 1.2 Guardrails

- The `.gitignore` already excludes `.env*` except `.env.example`. Keep it that way. Do not add `git add -f` on env files.
- When generating sample/test data, use fake credentials (e.g. `postgresql://test:test@localhost:5432/test`). Never seed real ones.
- When writing code that reads env vars, prefer `process.env.X!` with a clear runtime error if missing — never hardcode fallbacks to real values.
- Review your diff before committing. If a line contains `postgres://`, `sk-or-`, `sk-`, `Bearer `, a password, or a secret-looking string — it does not belong in the diff.

## 2. Design Laws (enforced by project structure)

The repository follows a strict layout. Place new code in the correct directory — do not invent new top-level folders.

```
src/
├── app/        # Next.js App Router: pages, layouts, route handlers (api/*)
├── components/ # Reusable React UI
│   ├── platform/   # App-specific, business UI tied to davajbacha domain
│   ├── reactbits/  # Third-party-style animated UI primitives
│   └── ui/         # Generic, headless/styled primitives (shadcn-style)
├── db/         # Drizzle ORM layer: client, schema, seed
├── hooks/      # Custom React hooks (client-side state)
└── lib/        # Server-side domain logic, integrations, pure modules
```

### 2.1 Routing & pages — `src/app/`

- One folder per route segment. Co-locate `page.tsx`, `layout.tsx`, `loading.tsx`, `error.tsx`, `not-found.tsx` inside the segment.
- API endpoints live in `src/app/api/<route>/route.ts` and are **thin**: parse input, call into `src/lib/`, return a `Response`. No business logic, no DB access directly in route handlers.
- Server Components by default. Add `"use client"` only when state, effects, or browser APIs are required.
- Cross-cutting concerns (auth checks, CSRF, rate limiting) live in `src/lib/` and are composed inside route handlers / server actions — not duplicated inline.

### 2.2 Components — `src/components/`

- **`platform/`** — Davajbacha-specific UI (dashboards, footprint UI, scenarios). May import from `ui/` and `lib/`. Never import from `app/`.
- **`ui/`** — Generic, reusable primitives (buttons, inputs, dialogs). **No business knowledge, no domain types, no DB calls.** Props are primitive or generic.
- **`reactbits/`** — Third-party / animated visual primitives. Treat as a vendor folder: do not modify their public API; wrap if you need behavior changes.
- A component belongs in `ui/` only if it would be equally useful in a different app. If it imports a domain type from `src/lib/platform-types.ts` or mentions "footprint", "scenar", "vstup" by name — it goes in `platform/`.

### 2.3 Database — `src/db/`

- `client.ts` — single Drizzle client instance. Imported wherever a DB connection is needed.
- `schema.ts` — Drizzle table definitions, one section per domain entity. Schema is the single source of truth for table shape.
- `seed.ts` — dev/test seeding only. Must be idempotent and safe to re-run.
- Migrations live in `drizzle/` (generated by `pnpm db:generate`). Never hand-edit generated migration SQL unless absolutely required; if you do, document why in the file header.
- **All** DB reads/writes from the app go through repository functions in `src/lib/platform-repository.ts` (or a sibling module in `lib/`). Route handlers and components must **not** import the Drizzle client directly.

### 2.4 Hooks — `src/hooks/`

- File name is `use-<thing>.ts` (kebab-case). Hook name is `useThing`.
- Client-side only. If it runs on the server, it does not belong here.
- Keep hooks small and single-purpose. If a hook grows past ~80 lines, split it.

### 2.5 Domain logic & integrations — `src/lib/`

- `platform-types.ts` — shared TypeScript types and DTOs for the platform domain. Imported by both client and server.
- `platform-repository.ts` — the **only** module that talks to `src/db/client.ts`. All queries live here. Functions take typed inputs and return typed rows; no `any`.
- `platform-data.ts` — static reference data, lookups, scoring tables. Pure data, no side effects.
- `scoring.ts` / `footprint-verification.ts` — pure domain algorithms. Must be deterministic and unit-tested (see `*.test.ts` siblings).
- `auth.ts` — session and authentication helpers. The only place that handles `AUTH_SECRET` and `ADMIN_PASSWORD`.
- `csrf.ts` — CSRF token issuance and verification.
- `rate-limit.ts` — request throttling. Used by route handlers.
- `openrouter.ts` — LLM API client. The only module that reads `OPENROUTER_API_KEY` / `OPENROUTER_MODEL`.
- `footprint-client.ts` — client-side wrapper for the footprint verification API.
- `utils.ts` — tiny generic helpers (`cn`, formatters). No domain knowledge.
- New integrations get their own file in `lib/` and follow the same pattern: one module, one external concern, env reads isolated to that module.

### 2.6 Cross-cutting rules

- **Imports flow downward only**: `app/` → `components/`, `lib/`, `hooks/`, `db/`. `lib/` may import from `db/`. `components/` may import from `lib/` and `hooks/`. `db/` and `hooks/` import nothing from `app/` or `components/`. No circular imports.
- **No business logic in components**. Components render and dispatch; `lib/` computes and persists.
- **No DB access in components**. Always go through `lib/platform-repository.ts`.
- **Env vars are read in `lib/` only**, at the boundary closest to the external system (`openrouter.ts` for LLM keys, `auth.ts` for auth, the DB client for `DATABASE_URL`). Components and route handlers receive typed values, never `process.env`.
- **Path alias**: use the project alias (`@/...`) for cross-folder imports; relative paths only within the same folder.

## 3. Conventions

- TypeScript strict. No `any` in new code. Prefer `zod` schemas at system boundaries (API input, env parsing).
- Files in `lib/` with pure logic must have a sibling `*.test.ts` (Vitest). Run `pnpm test` before pushing.
- Lint with `pnpm lint`, typecheck with `pnpm typecheck`. Both must pass.
- Commit messages: short imperative summary. No secrets in messages, branch names, or tags.

---

## 4. Quick checklist before any commit

- [ ] No `.env` file or real secret value in the diff
- [ ] No `DATABASE_URL`, API key, or password string in any added file
- [ ] `.env.example` still contains placeholders only
- [ ] New code lives in the correct folder per the design laws above
- [ ] `pnpm lint`, `pnpm typecheck`, `pnpm test` all pass
